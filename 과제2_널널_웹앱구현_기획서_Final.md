## 1. 과제 개요 및 선정 배경

지정과제 2번의 해결과제는 **"유명 관광지 중심의 관광객 쏠림 현상으로 인한 특정 명소의 오버투어리즘(과밀화) 문제"**다. 참고자료(붙임) 예시 서비스의 방향은 "실시간 데이터 기반 한적한 대안지 매칭으로 관광객 과밀을 분석해 여행 코스를 추천"하는 것이며, 관련 해시태그는 `#혼잡도 분석 #과밀지역 우회 #숨은 명소 #코스 추천` 4가지다. 예시는 개발 의무가 없는 참고용이지만, 1차 심사 시 이 해시태그와 연계된 기능에 대한 기능설명서 작성을 요청할 예정이므로, 이 4가지 축을 실제 구현 기능으로 구체화한다.

**해시태그 ↔ 구현 기능 매핑(기능설명서 골격)**

| 해시태그 | 구현 기능 | 근거 데이터 |
|---|---|---|
| #혼잡도 분석 | F3 널널도 분석(요일·시간대 분산 포함) | 관광지 집중률 예측, 빅데이터 방문자수, 수요 강도 |
| #과밀지역 우회 | F4 대안지 추천 + F6 추천 근거 카드 + F8 추천 분산 | 연관 관광지, 널널도 비교 |
| #숨은 명소 | F4 내 숨은 명소성 가점 로직 | 방문자수 하위 × 콘텐츠 풍부도 상위 교차(9-2) |
| #코스 추천 | F5 통합 코스 최적화 | 길찾기 API + 누적 혼잡도 |

> ⚠️ 유의: 동일 서비스로 ②-1에 이미 제출된 결과물이거나 ① 웹·앱 개발 부문과 동시 출품하는 서비스는 중복 제출로 심사 제외될 수 있다.

## 2. 서비스 정의

- **서비스명: 널널(Nullnull)**
  - "Null"(0, 텅 빔)과 "널널하다(여유롭다)"의 언어유희 — 붐비는 곳 대신 "지금 더 널널한" 곳을 찾아준다는 의미
  - 브랜드 메시지: `붐비는 곳 말고, 널널한 여행` / `유명한 곳의 매력은 유지하고, 혼잡은 줄이는 코스 추천` / `Null crowd, Full trip.`

**서비스 정의**
> 널널(Nullnull)은 한국관광공사의 위치기반·지역기반 관광정보(TourAPI)와 관광 빅데이터(집중률 예측·방문자수·수요 강도)를 활용해 인기 관광지의 과밀 위험을 방문 예정 시점 기준으로 분석하고, 유사한 매력의 한적한 대안 명소·코스를 추천함으로써 관광객 쏠림을 완화하는 분산형 여행 코스 추천 웹 서비스다.

**핵심 차별화: "3중 분산" 구조**

| 축 | 질문 | 대응 기능 |
|---|---|---|
| ① 시간 분산 | "같은 장소, 언제 가면 덜 붐빌까?" | 요일(±5일) + 당일 시간대(오전/오후/저녁) 비교 제안 (F3) |
| ② 공간 분산 | "비슷한 매력의 덜 붐비는 곳은 어디일까?" | 테마 유사 대안지·코스 추천 (F4·F5) |
| ③ 추천 분산 | "대안지가 새로운 과밀지가 되지는 않을까?" | 추천 노출 부하 기반 대안지 로테이션 (F8, 9-2) |

특히 ③은 "대안지 추천 서비스가 성공할수록 대안지가 다시 붐빈다"는 오버투어리즘 대응의 구조적 딜레마에 대한 답으로, 추천 자체를 분산시키는 로직을 서비스에 내장한다.

**경쟁·유사 서비스 대비 위치(신규)**

| 구분 | 혼잡도 표시 | 혼잡 "예측" | 대안 코스 자동 추천 | 추천 근거 수치화 | 추천 자체의 분산(2차 과밀 방지) |
|---|:-:|:-:|:-:|:-:|:-:|
| 포털 지도 서비스(네이버/카카오) | ○(일부 장소 실시간) | △ | ✕ | ✕ | ✕ |
| VisitKorea 등 공식 관광 정보 | △ | ✕ | ✕(코스는 편집형 콘텐츠) | ✕ | ✕ |
| 일반 여행 플래너 앱 | ✕ | ✕ | △(인기 기반) | ✕ | ✕ |
| **널널(Nullnull)** | ○ | ○(방문 예정일 기준) | ○(테마 유지형) | ○(감소율·유사도·이동시간) | **○(F8 로테이션)** |

> 기존 서비스는 "지금 붐빈다"까지 알려주지만, "그래서 어디로/언제로 옮기면 되는지"와 "옮긴 곳이 다시 붐비지 않게 하는 장치"는 없다. 널널은 이 두 칸을 채우는 서비스다. (비교표는 발표 전 각 서비스 최신 기능 재확인 후 확정)

**차별화 보조 문장**
> 널널의 추천은 "장소 나열"이 아니라 "혼잡을 낮추는 대체 가능성"의 수치화다. 각 추천 코스는 원 관광지 대비 혼잡 완화 효과, 테마 유사도, 이동 편의성을 함께 제시해 사용자가 납득 가능한 선택을 하도록 돕고, 방문 후 피드백으로 예측을 스스로 보정한다(9-4).

## 3. 문제 정의 및 시의성

> 기존 여행 정보는 인지도와 후기 중심으로 소비되어 특정 유명 관광지로 수요가 집중된다. 이는 ① 여행객에게는 대기·혼잡으로 인한 만족도 저하, ② 지역에는 특정 명소 인프라 과부하와 주민 정주권 침해, ③ 숨은 명소에는 기회 상실이라는 3중 손실을 만든다. 널널은 방문 예정일 기준 혼잡 예측과 지역 수요 데이터를 함께 분석해, 여행객의 만족도를 유지하면서 특정 명소의 과밀을 완화하고 수요를 시간·공간적으로 재배치하는 대안 코스를 제안한다.

**시의성 — 오버투어리즘은 이미 "규제"가 등장한 문제(신규)**

- 국내: 서울 북촌 한옥마을은 관광객 과밀로 인한 주민 정주권 문제가 심화되어, 종로구가 특별관리지역 지정과 함께 야간 시간대 관광객 방문 제한(통금) 정책을 시행하는 단계에 이르렀다. 오버투어리즘이 "불편" 수준을 넘어 **행정 규제로 대응해야 하는 사회 문제**가 되었다는 방증이다.
- 해외: 교토는 유명 관광지 쏠림 완화를 위해 후시미·오오하라 등 대안 지역을 알리는 관광객 분산 정책을 지속 추진해 왔다. 분산 전략 자체는 이미 정책적으로 검증된 접근이다.
- **널널의 포지션**: 규제(출입 제한)는 관광객 경험을 희생시키고, 수작업 홍보(대안지 캠페인)는 확장성이 없다. 널널은 **데이터와 알고리즘으로 자발적 분산을 자동화**한다 — "가지 말라"가 아니라 "여기가 더 좋다"로 문제를 푼다. 이 대조 구도를 PT 오프닝(15-2)에 사용한다.

> ※ 위 사례의 시행 시점·세부 조건은 발표자료 확정 전 최신 보도로 재확인한다.

## 5. MVP 핵심 기능

**시간 분산 → 공간 분산 → 추천 분산 → 방문 후 피드백 보정**으로 이어지는 순환 구조다. 아래 **8개 기능(F1~F8)**을 MVP로 확정하되, F7·F8은 로직·UI 부담이 작은 라이트 버전으로 구현해 2인 팀 일정을 지킨다. 일정 지연 시 축소 우선순위는 17장의 3단 사다리를 따른다.

| # | 기능 | 설명 | 관련 해시태그 |
|---|---|---|---|
| F1 | 스타일 프로파일링 | 여행 테마 태그(자연/역사/미식/포토스팟 등)·동행 유형 선택 → 테마 유사도 계산에 반영 | 공통 |
| F2 | 명소 검색 | 지역, 키워드, 카테고리 기반 관광지 검색 | 공통 |
| F3 | 널널도 분석(시간 분산) | 방문 예정일 혼잡도 5단계 표시 + 전후 ±5일 요일별 비교 + 당일 시간대(오전/오후/저녁) 비교(서울 MVP 시). **날짜 선택은 예측 데이터가 존재하는 오늘~+30일로 제한**(8-1) | #혼잡도 분석 |
| F4 | 대안지 추천(테마 매칭) | 스타일 태그 + 연관 관광지 데이터 기반, 같은 테마의 더 한적한 장소 추천 | #과밀지역 우회 #숨은 명소 |
| F5 | 통합 코스 최적화 | 대안지 2~4곳을 이동 동선과 코스 전체 누적 혼잡도를 함께 고려해 구성 | #코스 추천 |
| F6 | 추천 근거 카드 | 혼잡 감소율, 테마 유사도, 이동시간, 요일·시간대 이동 시 감소 효과 표시 | #과밀지역 우회 |
| F7 | 방문 후 피드백(라이트) | "생각보다 붐볐나요?" 1탭 피드백 수집 → 예측 보정 계수에 반영(9-4) | 지표 보정 |
| F8 | 추천 분산 — 2차 과밀 방지(라이트) | 최근 추천 노출·선택이 몰린 대안지에 점수 페널티를 부여해 대안지 풀을 로테이션. **로그가 없으면 페널티 0으로 자연 동작(콜드스타트 안전)**, 시드 계획은 9-2 | #과밀지역 우회 |

## 6. 사용자 흐름 / 데모 시나리오

심사 시연에서는 사용자가 1분 안에 서비스 가치를 이해해야 한다. **시연 스토리는 "시간 분산 → 공간 분산 → 근거 → 임팩트" 순서로, 3중 분산 구조가 자연스럽게 드러나도록 구성한다.**

1. 사용자가 `경복궁`과 `토요일 오후`를 입력(날짜 선택기는 오늘~+30일만 활성화, 기본값은 "다가오는 주말")
2. 널널도가 `매우 붐빔`으로 표시되고, 바로 아래에 "일요일 오전엔 `보통`, 같은 토요일이라도 저녁엔 `붐빔`" 시간 분산 제안이 함께 노출
3. 그래도 대안이 필요하면 "비슷하지만 더 널널한 코스 보기" 버튼 클릭
4. 대안 코스 2~3개가 카드와 지도에 표시
5. 각 코스에 `혼잡 42% 감소`, `이동 37분`, `역사문화 테마 유지` 근거 표시
6. 코스 선택 시 "이 선택으로 예상 혼잡 42%를 회피했어요" 임팩트 카드 노출
7. (시연 마무리 멘트) "이 대안지가 다시 붐비지 않도록, 널널은 추천 자체를 로테이션합니다" — F8 로직을 관리자 화면 수치로 짧게 보여줌

추천 근거 카드 문구 예시:
> 경복궁은 선택한 날짜에 `매우 붐빔`으로 예측됩니다. 널널은 같은 역사·문화 테마의 `서울한양도성 낙산구간`, `백인제가옥`, `운현궁`을 조합해 예상 혼잡도를 낮추고 이동시간을 40분 이내로 유지하는 코스를 추천합니다.

**UX 원칙**
- "추천합니다"보다 "왜 추천하는지"를 보여준다
- 원래 가려던 장소를 부정하지 않고 "같은 만족을 더 쾌적하게"라는 톤 유지
- 대안 제시 전에 시간 이동 옵션을 먼저 제시(가장 저항이 적은 분산)
- 혼잡도 수치의 출처와 예측 기준일을 작게 표시, 예측 불가 구간은 숨기지 않고 이유를 고지(데이터 정직성 원칙)

## 8. 데이터 및 API 설계

### 8-1. 예측 데이터의 범위와 표현 원칙

`관광지 집중률 방문자 추이 예측 정보`는 조회 시점 기준 **향후 30일 예측**만 제공된다. 여기서 두 가지 정책을 확정한다.

**(1) "실시간" 표현 구분**

| 구분 | 데이터 |
|---|---|
| 예측(핵심) | 관광지 집중률 방문자 추이 예측 정보, 빅데이터 지역별 방문자수, 지역별 관광 수요 강도 |
| 실시간(보조, 서울 집중 MVP 시) | 서울 실시간 도시데이터 API — 5분 단위 실시간 인구·혼잡도 + 향후 12시간 예측. F3 당일 시간대 분산에 활용 |

서비스 표현은 `실시간·예측 기반 혼잡도 판단`(서울 MVP) 또는 `방문 예정 시점 기준 혼잡 예측`(전국형)으로 통일한다.

**(2) 30일 예측 창(window) 처리 정책(신규 — 시연 결함 방지 최우선)**

| 구간 | 처리 |
|---|---|
| 오늘 ~ +30일 | 정상 예측 — 집중률 예측 API 기반 널널도 산출 |
| +30일 초과 | **MVP: 날짜 선택 UI에서 비활성화**(선택 자체가 불가) + "예측 데이터는 향후 30일까지 제공됩니다" 툴팁 고지 |
| (확장) +30일 초과 | 과거 `congestion_snapshot` 누적분 기반 요일·계절 패턴 추정치로 폴백, 뱃지에 `장기 추정` 라벨을 붙여 예측과 구분 표시 |

- MVP에서 UI 제한을 택하는 이유: 폴백 추정은 스냅샷이 수개월 누적되어야 신뢰할 수 있는데, 개발 기간(7~9월) 내 축적분으로는 근거가 약하다. 근거 없는 숫자를 보여주는 것보다 범위를 정직하게 제한하는 편이 "데이터 정직성" 원칙과 심사 대응 모두에 유리하다.
- 심사 질의 대응: "30일 이후는요?" → "예측 데이터 제공 범위를 UI에 그대로 반영했고, 스냅샷이 축적되면 요일·계절 패턴 기반 장기 추정으로 확장합니다(로드맵 5장)."

### 8-2. 한국관광공사 OpenAPI (필수 활용)

공공데이터포털·한국관광 콘텐츠랩(api.visitkorea.or.kr)에서 검색·신청 가능한 아래 5종을 조합한다. 팀 대표가 사전에 공공데이터포털 회원가입 및 개별 API 활용신청을 진행해야 한다.

| API | 제공 방식 | 제공 정보 | 추천 로직 내 역할(MVP) |
|---|---|---|---|
| 국문 관광정보 서비스(TourAPI) | 한국관광공사 | 지역기반/위치기반 관광정보, 키워드검색, 공통 상세정보, 이미지 정보, 분류체계 코드 | 검색·상세·지도 마커·후보지 마스터 데이터, 콘텐츠 풍부도(숨은 명소성) 산출 |
| 관광지 집중률 방문자 추이 예측 정보 | 통신사(KT) 데이터 기반 | 향후 30일간 관광지별 방문자 집중률 예측 | 널널도 산출의 핵심 지표, 요일 분산 비교 |
| 빅데이터 지역별 방문자수 | 통신사(KT·SKT) 데이터 기반 | 광역/기초지자체별 일자별 순방문자 수 | 같은 지역 전체가 붐비는지 보정, 숨은 명소성의 "방문자수 하위" 판정 |
| 관광지별 연관 관광지 정보 | 티맵 모빌리티 내비게이션 데이터 기반 | 관광지별 연관성 있는 주변 관광지 목록 | 대안 후보군 생성, 테마 유사도 결합 항 |
| 지역별 관광 수요 강도 | 한국관광공사 | 지역 단위 관광 수요 지표 | 지역 과밀 보정 및 숨은 지역 가점 |

> ⚠️ TourAPI에 사용자 평점·리뷰 필드가 있는지 여부를 포함해, 상세 요청 파라미터는 회원가입 후 콘텐츠랩 Swagger 문서에서 **개발 착수 전 반드시 재확인**한다. (문의: tourapi@knto.or.kr) 본 기획서의 숨은 명소성 산출은 평점 필드가 없다는 보수적 가정 하에 설계했다(9-2).

### 8-3. 보조 외부 API (MVP 조합 확정)

| API | MVP 포함 | 용도 |
|---|:-:|---|
| 카카오맵/네이버지도 SDK | ○ | 지도 시연 완성도 |

**MVP 조합 권장안**
- 전국형 MVP: 공사 OpenAPI 5종 + 카카오맵/네이버지도 + 기상청 (표현: "예측 기반 혼잡도", 시간 분산은 요일 단위)
- 서울 집중 MVP(기본안 권장): 공사 OpenAPI 5종 + 서울 실시간 도시데이터 + 카카오맵/네이버지도 + 기상청 (표현: "실시간·예측 기반 혼잡도", 당일 시간대 분산까지 시연 가능)

> 서울 집중 MVP를 택할 경우 시연 지역을 `서울 주요 관광지 베타`로 명확히 한정하고, 전국 확장은 로드맵(16장)으로 제시한다.

## 9. 널널도 지표 및 추천 알고리즘

### 9-1. 널널도(혼잡 위험) 지표

사용자에게는 복잡한 수식 대신 5단계 뱃지로 보여준다.

| 점수 | 상태 | UI 표현 |
|---:|---|---|
| 0~20 | 매우 널널 | 초록 |
| 21~40 | 널널 | 연두 |
| 41~60 | 보통 | 노랑 |
| 61~80 | 붐빔 | 주황 |
| 81~100 | 매우 붐빔 | 빨강 |

내부 산식(가중치는 초기값이며 9-5 백테스트로 검증·조정, 설정 파일로 외부화):
```
CongestionRisk =
  0.55 × 관광지 집중률 예측값
+ 0.20 × 지역 방문자수 상대지수
+ 0.15 × 지역 관광 수요 강도
+ 0.10 × 요일/공휴일/날씨 보정값
```
- **날씨 보정은 방문일이 기상청 단기예보 범위 내일 때만 적용**하고, 범위 밖이면 해당 항을 제외한 뒤 남은 가중치를 재정규화한다(합계 1.0 유지). 산식에 넣어 놓고 연동은 "선택"으로 두는 불일치를 제거.
- 서울 집중 MVP에서 당일 시간대 조회 시, 서울 실시간 도시데이터의 12시간 예측을 집중률 예측값 자리에 대입해 시간대별 CongestionRisk를 산출한다.

### 9-2. 대안지 추천 점수

```
AlternativeScore =
  0.30 × 테마 유사도
+ 0.25 × 혼잡 완화 효과
+ 0.15 × 이동 편의성
+ 0.10 × 숨은 명소성
+ 0.10 × 날씨/운영 적합성
− 0.10 × 추천 부하(RecommendationLoad)
```

| 항목 | 정의(v4 구체화) |
|---|---|
| 테마 유사도 | `0.6 × Jaccard(카테고리 코드 집합A, 집합B) + 0.4 × 연관 관광지 API similarity(정규화)`. TourAPI 대/중/소분류 코드를 집합으로 취급해 자카드 유사도를 계산하고, 연관 관광지 API가 제공하는 연관도 수치를 결합. 두 소스가 모두 없으면 키워드 매칭으로 폴백 |
| 혼잡 완화 효과 | 원 관광지 CongestionRisk − 대안지 CongestionRisk |
| 이동 편의성 | 원 관광지 또는 숙소 기준 이동시간 |
| 숨은 명소성(재정의) | **TourAPI 실제 필드 기반 프록시**: `방문자수 하위 분위(빅데이터 방문자수 기준) × 콘텐츠 풍부도 상위`. 콘텐츠 풍부도 = 이미지 개수 + 개요(overview) 텍스트 길이 + 연관 관광지 목록 내 등장 횟수의 정규화 합. "평판/평점" 같은 미확인 필드에 의존하지 않음 |
| 날씨/운영 적합성 | 야외/실내 구분, 영업시간, 기상 조건(예보 범위 내 조건부, 9-1과 동일 원칙) |
| 추천 부하 | 최근 7일간 해당 대안지의 추천 노출 횟수 + (선택 횟수 × 2)를 0~1로 정규화. `recommendation_log` 집계 |

**F8(추천 분산)의 콜드스타트 대응(신규)**

1. **Graceful degradation**: 로그가 없으면 RecommendationLoad = 0 → 페널티 없이 나머지 5개 항으로 정상 추천된다. F8은 데이터가 없어도 서비스가 깨지지 않는 구조다.
2. **시드 데이터 계획(2단계)**
   - 1차(자연 시드): 8월 개발·QA 시나리오 자체를 로그 적재 겸용으로 설계 — 팀 2인의 반복 테스트만으로도 노출 로그의 기본 분포가 생긴다.
   - 2차(베타 시드): 9월 초 지인 베타 테스터 10~20명에게 "경복궁·N서울타워·홍대 등 대표 명소 5곳 조회 → 대안 코스 1개 선택" 시나리오를 배포해 노출·선택 로그를 축적한다. 이는 F7 피드백 시드도 겸한다.
   - 보조(합성 시드): 관리자 전용 시딩 배치로 합성 로그를 주입할 수 있게 하되, **합성분은 `is_seed=true`로 구분 저장**하고 시연·발표에서 실로그와 합성로그를 구분해 고지한다(데이터 정직성 원칙 — 합성 데이터를 실사용 실적처럼 보이게 하지 않는다).
3. **시연 프레이밍**: 로그가 충분하면 실데이터 로테이션을 시연하고, 부족하면 "로직·코드 완성 + 데이터 축적 단계"로 프레이밍하며 시드 로그 기반 동작 화면을 보여준다. 어느 쪽이든 시연이 성립하도록 이중화.

### 9-3. 코스 점수 (동선 반영)

```
CourseScore =
  평균 AlternativeScore
− 이동시간 패널티
− 동일 카테고리 반복 패널티
+ 지역 분산 보너스
```

### 9-4. 피드백 기반 예측 보정

F7에서 수집한 1탭 피드백("생각보다 붐볐다 / 예상대로 / 생각보다 한산했다")을 장소별 보정 계수로 환산해 예측 오차를 줄인다.

```
adjusted_risk(spot, t) = CongestionRisk(spot, t) × (1 + α × feedback_bias(spot))

feedback_bias(spot) = EWMA( 체감 혼잡(−1/0/+1) 의 최근 30건 )
α = 0.2 (초기값, 튜닝 대상)
```

- 피드백 30건 미만 장소는 보정 미적용(콜드스타트 왜곡 방지). 베타 시드(9-2)로 대표 명소부터 임계치 도달을 노린다.
- 보정은 일 1회 배치로 `spot_score_daily`에 반영 → 실서비스 조회 시 추가 연산 없음.

### 9-5. 가중치 검증 — 백테스트 계획(신규)

"왜 이 가중치인가"에 대한 근거를 만들기 위해, 8월 실데이터 확보 직후 1회 백테스트를 수행한다.

| 단계 | 내용 |
|---|---|
| 검증 세트 | "확실히 붐빌 것"이 알려진 날(주말·공휴일·축제일)과 평일을 섞은 대표 명소 10곳 × 14일 격자 |
| 검증 방법 | 초기 가중치로 산출한 CongestionRisk가 상식적 순서(공휴일 > 주말 > 평일, 대표 명소 > 인근 소규모 명소)를 재현하는지 순위 상관으로 확인 |
| 조정 | 순서가 깨지는 구간의 항목 가중치를 ±0.05 단위로 조정, 최종값을 설정 파일(`weights.yaml`)로 확정 |
| 산출물 | 백테스트 전/후 비교 1페이지 → 기능설명서·발표자료에 "데이터로 검증한 가중치"로 인용 |

가중치를 코드가 아닌 설정 파일로 외부화하면, 심사 질의 시 "가중치는 검증 절차를 거친 조정 가능한 파라미터"라고 답할 수 있고 향후 지역별 가중치 차등화(확장)의 기반도 된다.

이 알고리즘 체계(널널도 → 대안지 점수(추천 분산 포함) → 코스 점수 → 피드백 보정 → 가중치 검증)는 "서비스 기획력(독창성)"·"데이터 활용 적절성" 배점에 직접 대응하므로, 1차 심사 기능설명서에 수식·판단 기준·검증 결과를 그대로 명시한다.

## 10. 시스템 아키텍처 (BE·AI·Infra 담당)

시연/1차 심사 목표는 인프라 복잡도가 아니라 **완성 서비스의 구동 안정성**이므로, MVP는 단순 구성으로 먼저 완성하고, AWS 기반 확장 아키텍처는 "서비스 발전성" 항목 대응용 로드맵으로 별도 제시한다.

### 10-1. MVP 아키텍처 (구현 우선순위 1)

| 레이어 | 권장 구성 |
|---|---|
| Frontend | React/Vite, S3+CloudFront 또는 Vercel 배포 |
| Backend | FastAPI 또는 Spring Boot 단일 API 서버(추천 엔진 로직 포함, 가중치 설정 파일 로드) |
| DB | PostgreSQL 1개(RDS 프리티어 또는 단일 인스턴스) |
| Batch | GitHub Actions cron / Lambda / 서버 내 스케줄러 중 택1 — ① 공사 OpenAPI 수집, ② 피드백 보정 반영, ③ 추천 부하 집계를 하나의 일배치로 통합 |
| Cache | 별도 Redis 없이 DB 캐시 테이블(`spot_score_daily`)로 시작 |
| Map | 카카오맵/네이버지도 SDK |
| Demo | `congestion_snapshot` 기반 오프라인 데모 모드(`?demo=true`) |

### 10-2. 확장 아키텍처 (서비스 발전성 로드맵, 2단계)

| 레이어 | 확장 구성 |
|---|---|
| API 서버 | ECS Fargate + ALB로 분리, 추천 엔진을 Lambda로 별도 분리 |
| Cache | ElastiCache Redis로 전환(API 응답·혼잡도 캐시 TTL 관리) |
| Batch | EventBridge Scheduler + Lambda로 정례화 |
| IaC/배포 | Terraform 또는 AWS CDK로 인프라 코드화, GitHub Actions → ECR → ECS 배포 |
| 운영 | CloudWatch 모니터링·알람, Secrets Manager로 API 키 관리, Route53+ACM |

> 팀원이 AWS SAA/DVA 보유자이므로, **확장 아키텍처 중 핵심 모듈(VPC·ECS 서비스 정의·EventBridge 배치)의 Terraform/CDK 코드를 실제 작성해 공개 저장소에 올려둔다**(배포는 생략 가능). "설계만 한 게 아니라 코드까지 준비했다"는 실물 근거로, 발표자료에 저장소 링크와 코드 스니펫 1장을 포함한다.

## 11. 데이터베이스 설계 (ERD 개요)

| 테이블 | 주요 컬럼 | 설명 |
|---|---|---|
| `tourist_spot` | spot_id(PK), content_id, name, category, addr, lat, lng, image_url, image_count, overview_len | TourAPI 동기화 대상(콘텐츠 풍부도 필드 추가) |

| `related_spot` | spot_id(FK), related_spot_id(FK), similarity_score | 연관 관광지 정보 캐시 |
| `course` | course_id(PK), title, region, base_spot_id(FK), created_at | 생성된 추천/저장 코스 |
| `course_item` | course_id(FK), spot_id(FK), order_no, reason_text | 코스 구성 장소 및 추천 이유 |
| `spot_score_daily` | spot_id(FK), date, congestion_risk, adjusted_risk, alternative_score | 장소별·날짜별 점수 캐시(보정치 포함) |
| `recommendation_evidence` | course_id(FK), spot_id(FK), theme_sim, relief_effect, travel_time, hidden_score | 추천 근거 카드 표시 수치 |
| `recommendation_log` | id(PK), spot_id(FK), exposed_at, selected(bool), **is_seed(bool)** | 대안지 추천 노출·선택 기록 → 추천 부하 산출. 합성 시드 구분 컬럼 추가 |
| `visit_feedback` | id(PK), course_id(FK), spot_id(FK), perceived(-1/0/+1), created_at, **is_seed(bool)** | 방문 후 1탭 피드백 → 보정 계수(9-4) |
| `api_ingest_log` | api_name, status, last_synced_at, error_message | 공사 API 수집 상태 기록(구동 안정성 근거) |
| `user`(선택) | user_id(PK), nickname | MVP는 로그인 없이 익명 코스 생성 |
| `bookmark`(확장) | user_id(FK), course_id(FK) | 코스 저장 기능용 |

분산 임팩트 집계는 별도 테이블 없이 `recommendation_log`·`course_item`·`spot_score_daily` 조인 집계 뷰(주간)로 구현하며, 집계 시 `is_seed=true` 데이터는 제외한다.

## 16. 차별화 및 확장 로드맵

### 16-1. 분산 효과 지표
추천 결과에 오버투어리즘 완화 효과를 정량 제시한다: 원 관광지 대비 예상 혼잡도 42% 감소, 동일 테마 유지율 87%, 이동시간 38분 이내, 과밀 지역 밖 대안지 2곳 포함 등. 여기에 **주간 집계 임팩트(숨은 명소 선택 N회, 평균 회피율 M%)**를 메인에 노출해 개인 편익을 사회적 가치로 확장한다.

### 16-2. 지자체/관광공사 확장 가능성
- 관광객용: 널널 코스 추천 (MVP)
- 지역 운영자용: 과밀 관광지-대안 관광지 연결 현황 + 추천 부하 분포 대시보드 (확장) — `recommendation_log` 데이터가 그대로 정책 근거 데이터가 됨
- 지자체용: 분산 유도 캠페인 후보지 추천 (확장)
- **PT용 실물 근거(신규)**: 위 대시보드를 가상 데이터 기반 **목업 스크린샷 1장**으로 제작해 발표자료에 포함한다(9월 4주차 WBS 반영). 구현 없이도 "발전성"을 구체적 화면으로 보여주는 최소 비용 장치다. 목업임을 명시해 실구현으로 오인되지 않게 한다.

### 16-3. 다국어 전략
> MVP는 한국어 웹으로 구현하되, TourAPI 외국어 관광정보와 브라우저 언어 설정을 활용해 영어·일본어·중국어 관광객 대상 확장을 고려한다.

## 17. 리스크 및 대응방안

| 혼잡도 데이터가 예측치이며 실시간이 아님 | "예측 기반" 문구 명시, 서울 MVP 시 실시간 도시데이터 보강, 피드백 보정으로 오차 축소 |

| 심사장 네트워크·외부 API 장애 | `?demo=true` 오프라인 스냅샷 모드 사전 준비 및 리허설 |
| 추천 로그·피드백 콜드스타트 | QA 겸 자연 시드 → 베타 테스터 시드 → 합성 시드(is_seed 구분) 3단 대응(9-2), 시연 프레이밍 이중화 |
| 2인 팀 리소스 한계·일정 지연 | **기능 축소 3단 사다리 사전 합의**: ① 분산 임팩트 집계 카운터(메인) 제외 → ② F3 당일 시간대 분산 제외(요일 비교까지만 유지) → ③ F8 라이브 동작 제외(로직·코드는 완성, "데이터 축적 단계"로 발표). F8은 독창성 핵심이자 구현 비용이 가장 작으므로 최후순위로 배치하고, 실제 리소스가 큰 실시간 연동·집계부터 축소한다 |
| 대회 규정(중복 출품 등) 위반 리스크 | 접수 전 유의사항 재확인, ①·②-1 제출작과 결과물 동일성 여부 사전 점검 |
| 심사 시점까지 배포 미흡 | 1차 심사는 개발 완료된 완성 서비스 대상이므로, 8월 중 조기 배포 후 지속 보완 |
